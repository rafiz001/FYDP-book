\chapter{System Architecture}
This chapter contains the general description of the proposed system. It covers the system requirements, architecture, database design, and technologies used to develop the application.

\section{Requirement Analysis}
Requirement Analysis defines what the proposed system is expected to do and how it is supposed to perform to fulfill the stated objectives. It serves as the basis for system design and implementation because it highlights what the proposed system does, its behavior, and the quality attributes expected of the system.

The proposed system, A Cross-Platform Multi-Vendor Pharmacy ERP System with AI-Based Stock Forecasting and Analytics, is intended to automate the management of various departments in multiple pharmacies. The system is designed to manage stock, purchases, sales, suppliers, customers, employees, finances, reports, and notifications in a cross-platform environment while enforcing the role-based management of the system functions. The system will also utilize artificial intelligence to analyze previous sales and stock data to forecast future medicine needs, highlight potential shortages and expiry risks, and recommend the quantities to purchase for the different pharmacies.

\section{Detailed Methodology and Design}
The proposed A Cross-Platform Multi-Vendor Pharmacy ERP System with AI-Based Stock Forecasting and Analytics adopted the client-server architecture \cite{xu2020system}, which ensured scalability, operability, and effective communication of the system’s components. The system comprised three main parts: frontend, backend, and database. The frontend included the website, which was built on NextJS framework, and the mobile application, which was developed in Flutter framework. The website and the mobile app used REST API for effective communication with the backend.

The application backend was built with Express.js, which handled the request and response logic in the system. It also managed such aspects as user authentication and authorization, request validation, business logic, and pharmacy operation. Additionally, the backend accommodated the forecasting system that used AI to predict demand based on past sales and inventory data, while also presenting warnings on low stock, expiry dates, and recommended items for reorder.

The MongoDB database was utilized to store all the information related to the app, such as users, medicines, inventory, purchases, sales, suppliers, customers, employees, and AI forecasts. Additionally, the Redis database was applied as an in-memory cache to enhance the performance of the application by decreasing the number of requests to the MongoDB database.

The general process started when a request from the user came to the web or the mobile application. The application then relayed the request to the backend, where it was authenticated and processed. The backend accessed the database to retrieve or update the required information according to the request type and returned the result to the application. In case of forecasts, the backend relayed the sales and inventory data to the AI, processed the results, stored them in the database, and displayed the forecast results in the dashboard.
